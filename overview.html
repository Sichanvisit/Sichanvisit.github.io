Obsidian RAG 01 Problem

검색은 되지만 지금 읽는 노트 기준의 다음 흐름은 보이지 않았습니다

기존 문서형 RAG는 답변은 만들 수 있었지만, 현재 노트와 연결된 근거와 다음 행동까지 한 번에 이어주지는 못했습니다.

배경

V1은 Streamlit 중심 실험 환경에 가까워서, 실제 Obsidian 작업 문맥 안에서 바로 쓰기에는 동선이 멀었습니다.

실제 불편

현재 note와 linked note를 따로 열어 비교해야 해서 읽기와 작업이 자주 끊겼고, 답변 뒤에 후속 행동도 이어지지 않았습니다.

왜 어려웠나

relation 정보가 retrieval에 직접 반영되지 않아 지금 문맥 기준의 확장 검색, 추천, 우선순위 판단이 약했습니다.

해결 목표

현재 문서를 기준으로 근거 문서, 추천 문서, 다음 액션이 같은 화면에서 이어지는 작업형 RAG를 만드는 것이 목표였습니다.

채용 포인트

문제를 단순한 챗봇 성능 이슈가 아니라, 실제 사용 흐름과 UX가 끊기는 구조 문제로 다시 정의한 사례입니다.

Obsidian RAG 02 Approach

현재 문맥, relation retrieval, 추천 액션을 하나의 인터페이스로 묶었습니다

플러그인 메인 클라이언트와 Ops를 분리하고, relation runtime을 통해 현재 문서 주변의 연결 노트를 함께 가져오도록 다시 설계했습니다.

내 역할

FastAPI backend, plugin main client, relation runtime, recommendation/action payload를 직접 설계하고 구현했습니다.

핵심 설계 1

note, linked note, typed relation schema를 retrieval 기준으로 결합해 relation-aware retrieval 흐름을 만들었습니다.

핵심 설계 2

검색 결과를 추천 payload와 action-ready 구조로 확장해, 답변 이후 행동까지 시스템 안에서 이어지게 했습니다.

선택 이유

답변만 보여주면 사용자가 다시 탐색을 시작해야 해서, source와 recommendation, next action을 같은 화면 흐름에 놓았습니다.

채용 포인트

기능을 더한 것이 아니라, 문서 문맥과 후속 행동이 이어지는 전체 사용자 흐름을 기준으로 구조를 다시 설계했습니다.

Obsidian RAG 03 Outcome

답변형 도구가 아니라 실행 중심의 지식 워크스페이스로 정리했습니다

핵심 결과는 검색 품질 자체보다도, 답변과 근거 문서, 추천 문서, 다음 액션이 하나의 작업 흐름으로 이어지게 만든 점입니다.

결과

검색 결과를 읽고 끝나는 구조가 아니라, 후속 문서 탐색과 다음 작업까지 이어지는 흐름을 만들었습니다.

설명 가능성

현재 문서 중심 retrieval과 recommendation 구조를 시연할 수 있어, 단순 챗봇보다 훨씬 명확하게 설명 가능합니다.

제품 관점

Obsidian 안에서 실제로 쓰는 로컬 지식 워크플로로 정리되며, 도구가 아니라 제품형 워크스페이스 경험으로 보이게 됐습니다.

배운 점

RAG 품질 개선만으로는 부족하고, schema 설계와 action 설계가 함께 가야 실제 사용성이 생긴다는 점을 확인했습니다.

채용 포인트

RAG 성능 튜닝만 한 프로젝트가 아니라, 실제 사용 흐름을 설계한 로컬 지식 워크스페이스 사례로 소개할 수 있습니다.

Generative Pipeline 01 Problem

부분 수정이 생겨도 전체를 다시 돌려야 해서 비용과 실패 리스크가 컸습니다

상세페이지 제작은 실제로는 한 장만 바꾸고 싶은 경우가 많은데, 기존 흐름은 일부 수정에도 전체 생성 파이프라인을 다시 태워야 했습니다.

배경

상세페이지 제작은 기획, 배경 생성, 후처리, 문구 배치, 수정 반복이 계속 이어지는 작업이라 일회성 생성으로 끝나지 않습니다.

실제 불편

기획, 생성, 편집이 한 덩어리로 묶여 있어서 수정 포인트가 생겨도 어디를 다시 돌려야 하는지 분리하기 어려웠습니다.

왜 어려웠나

생성 단계 하나가 실패하면 전체 작업이 다시 커지고, 병목과 실패 지점을 단계별로 설명하기도 어려웠습니다.

해결 목표

예쁜 결과 몇 장보다도, 부분 수정과 실패 이후를 버틸 수 있는 운영 가능한 제작 파이프라인을 만드는 것이 핵심 목표였습니다.

채용 포인트

생성 AI를 결과물 데모가 아니라, 반복 수정과 실패 리스크를 다루는 운영 구조 문제로 재정의한 사례입니다.

Generative Pipeline 02 Approach

Director와 5-stage 파이프라인으로 책임 분리와 복구 경로를 만들었습니다

기획, 배경 생성, 업스케일, 보정, 광고 조합을 나누고 failover와 page regenerate를 넣어 서비스형 생성 흐름으로 다시 구성했습니다.

내 역할

FastAPI orchestration, Director, 5-stage pipeline, LLM failover, regenerate 경로를 직접 설계했습니다.

핵심 설계 1

기획과 생성을 분리하고, manifest 기반으로 초기화와 실행을 구분해 수정 지점을 명확하게 만들었습니다.

핵심 설계 2

segmenter부터 ad designer까지 5-stage로 나누고, 단계별 시간 기록과 page regenerate 경로를 넣었습니다.

선택 이유

실무에서는 전체 재생성보다 부분 수정과 재시도가 더 자주 일어나기 때문에, 복구 가능한 구조가 꼭 필요했습니다.

채용 포인트

모델 조합보다 먼저 시스템 책임을 나누고 재생성 경로를 설계했다는 점이, 서비스형 AI를 다루는 역량으로 연결됩니다.

Generative Pipeline 03 Outcome

결과 이미지보다 운영 가능한 생성 시스템 설계 경험으로 정리했습니다

핵심 결과는 예쁜 산출물 몇 장보다도, 생성 이후 수정과 재시도를 다루는 실제 제작 파이프라인을 설계한 점에 있습니다.

결과

부분 수정과 페이지 단위 재생성이 가능한 실무형 제작 흐름으로 설명할 수 있는 구조를 만들었습니다.

설명 가능성

병목과 실패 지점을 단계별로 추적하고 공유할 수 있어, 결과물뿐 아니라 운영 과정도 함께 설명 가능합니다.

서비스 관점

결과 이미지 데모보다 서비스형 생성 파이프라인을 설계한 경험이 더 선명하게 드러나게 됐습니다.

배운 점

생성 AI도 결국 운영 시스템이기 때문에, 오케스트레이션과 실패 대응 구조가 품질만큼 중요하다는 점을 확인했습니다.

채용 포인트

생성 AI를 사용한 경험이 아니라, 생성 이후 수정과 복구를 포함한 전체 서비스 파이프라인을 설계한 경험으로 보여줄 수 있습니다.

CFD Trading 01 Problem

데이터, 판단 근거, 운영 화면이 분리되면 설명과 검증이 어려웠습니다

시장 데이터 수집, 판단 로직, 운영 확인 화면이 따로 움직이면 실제로 어떤 근거로 의사결정을 했는지 추적하기가 어려워집니다.

배경

트레이딩에서는 가격 예측 자체보다, 데이터를 어떤 상태로 해석하고 있는지와 그 판단을 운영자가 다시 읽을 수 있는지가 중요합니다.

실제 불편

시장 데이터와 판단 로직, 운영 확인 화면이 분리되면 분석 결과가 실제 의사결정으로 어떻게 이어졌는지 보기 어렵습니다.

왜 어려웠나

규칙 기반 판단과 ML 보조 판단이 따로 놀고, API와 대시보드도 분리되면 설명 가능성과 검증 흐름이 약해집니다.

해결 목표

예측 모델 하나보다도, 데이터 해석부터 운영 검증까지 같은 언어로 이어지는 설명 가능한 시스템을 만드는 것이 목표였습니다.

채용 포인트

예측 정확도 중심이 아니라, 데이터 해석부터 운영 검증까지 이어지는 설명 가능한 시스템 문제로 접근했습니다.

CFD Trading 02 Approach

정규화 파이프라인과 상태 해석 레이어를 중심으로 흐름을 다시 묶었습니다

Market·MT5 데이터를 정규화하고, 상태 해석을 기준으로 규칙 판단과 ML 보조 판단, 운영 API와 대시보드를 하나의 구조로 연결했습니다.

내 역할

데이터 처리 파이프라인, state interpretation, ML assist 흐름, FastAPI runtime API, Next.js dashboard를 직접 연결했습니다.

핵심 설계 1

정규화된 데이터와 상태 해석을 공통 언어로 두고, 분석과 판단, 운영 화면이 같은 기준을 보도록 정리했습니다.

핵심 설계 2

규칙 기반 판단 위에 ML assist를 붙이고, runtime·trades·ops API와 대시보드가 같은 상태 정보를 보도록 연결했습니다.

선택 이유

운영 단계에서 근거가 바로 보여야 실제로 검증과 설명이 가능해지기 때문에, API와 대시보드까지 같은 흐름으로 묶었습니다.

채용 포인트

데이터 엔지니어링, 해석 로직, 운영 API, 대시보드를 따로 만들지 않고 하나의 설명 가능한 시스템으로 연결한 사례입니다.

CFD Trading 03 Outcome

예측 모델보다 설명 가능한 트레이딩 분석 시스템으로 정리했습니다

같은 데이터를 분석, 판단, API, 대시보드에서 재사용하며 운영 화면에서 근거를 함께 보여줄 수 있는 구조가 됐습니다.

결과

분석, 판단, API, 대시보드가 같은 데이터 기준으로 이어지며 상태와 판단 근거를 운영 화면에서 확인할 수 있게 됐습니다.

설명 가능성

시장 상태, 규칙 판단, ML 보조 판단을 한 흐름에서 보여줄 수 있어 시연과 검증이 쉬워졌고 포트폴리오 설명력도 높아졌습니다.

운영 관점

대시보드와 API, 문서, 테스트를 함께 두어 운영자가 시스템 상태를 빠르게 확인하고 변경 영향도 파악할 수 있게 했습니다.

Research Foundation

이 설명은 research 섹션에 쌓인 노트를 바탕으로 다시 묶은 분석 흐름입니다

여기서 말하는 데이터 분석 방식은 새로 덧붙인 소개 문장이 아니라, 네가 이미 정리해 둔 research 아카이브에서 반복적으로 드러난 기준을 다시 꺼내어 보이게 한 것입니다. 문제를 정의하고, 데이터 표현을 정하고, 지표를 고르고, 실패 케이스를 기록한 뒤, 그 결과를 다시 읽히는 자료로 정리하는 흐름이 portfolio 프로젝트에도 그대로 이어집니다.

Research Note -> Project Metric -> Error Pattern Archive -> Design Decision

Machine Learning Sprint Archive에 쌓인 회귀, 분류 실험은 어떤 모델을 썼는지보다 어떤 지표를 먼저 볼지 정하는 기준을 남깁니다. RMSLE, Recall, F1, ROC-AUC처럼 문제마다 실패를 읽는 기준이 다르다는 점이 이후 프로젝트의 baseline 판단 방식으로 이어졌습니다.

How I Analyze Data

데이터를 이렇게 분석하고 정리합니다

ML Workflow

모델보다 먼저 데이터 단위, 지표, 실패 케이스를 정리합니다

회귀, 분류, 문서형 RAG를 같은 방식으로 다루지 않습니다. 먼저 예측 대상과 입력 표현을 정의하고, EDA로 변수와 구조를 확인한 뒤, 문제에 맞는 지표와 오류 패턴을 기준으로 다음 실험을 설계합니다. 그 과정은 노트와 포트폴리오에 다시 정리해 다음 프로젝트의 판단 근거로 이어갑니다.

EDA -> Feature -> Metric Raw + Summary + Metadata Error Note -> Next Experiment
질문과 지표를 먼저 정의합니다

문제 유형에 맞는 기준이 먼저 정해져야 실험이 흔들리지 않습니다. 회귀에서는 RMSLE 같은 오차 기준을, 불균형 분류에서는 Recall, F1, ROC-AUC처럼 놓치면 안 되는 실패 패턴을 먼저 보고 baseline과 feature를 조정합니다.

{% if ml_archive %} ML Sprint {% endif %} {% if cfd %} CFD Case {% endif %}
데이터 표현을 다시 설계합니다

문서형 데이터는 수집보다 표현 방식이 더 중요합니다. 원문, 요약, 표, 메타데이터를 분리하고 retrieval-friendly representation으로 재구성해 검색 품질과 응답 근거를 함께 잡습니다.
