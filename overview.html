Obsidian RAG 01 Problem

검색은 되지만 지금 읽는 노트 기준의 다음 흐름은 보이지 않았습니다

V1에서는 답변은 만들 수 있었지만, 지금 읽는 note를 기준으로 근거 문서와 다음 행동까지 한 흐름으로 이어지지 않았습니다. 그래서 이 프로젝트는 챗봇 하나를 더 만드는 일이 아니라, 읽기와 실행이 끊기지 않는 지식 작업 화면을 다시 정의하는 일에서 시작했습니다.

배경

V1은 Streamlit 중심 실험 환경에 가까워서 retrieval 품질을 확인하기에는 좋았지만, 실제 Obsidian 작업 문맥 안에서 바로 쓰기에는 동선이 멀었습니다. 사용자는 노트를 읽다가도 별도 콘솔로 이동해 질문하고 다시 돌아와야 했습니다.

실제 불편

현재 note와 linked note를 따로 열어 비교해야 해서 읽기와 작업이 자주 끊겼고, 답변 뒤에 후속 행동도 이어지지 않았습니다. 결국 사용자는 답을 본 뒤 다시 파일을 찾고, 다음 액션을 스스로 정리해야 했습니다.

왜 어려웠나

relation 정보는 vault 안에 있었지만 retrieval에 직접 반영되지 않아 지금 문맥 기준의 확장 검색, 추천, 우선순위 판단이 약했습니다. 문서 간 연결이 있어도 시스템은 그 연결을 작업 흐름으로 읽어내지 못했습니다.

해결 목표

현재 문서를 기준으로 근거 문서, 추천 문서, 다음 액션이 같은 화면에서 이어지는 작업형 RAG를 만드는 것이 목표였습니다. 검색 정확도보다도, 읽기에서 실행으로 이어지는 UX를 복구하는 것이 더 중요한 문제라고 판단했습니다.

채용 포인트

문제를 단순한 챗봇 성능 이슈가 아니라, 실제 사용 흐름과 UX가 끊기는 구조 문제로 다시 정의한 사례입니다. 채용관 입장에서는 기능 추가보다 문제 재정의 능력이 먼저 보이는 카드입니다.

Obsidian RAG 02 Approach

현재 문맥, relation retrieval, 추천 액션을 하나의 인터페이스로 묶었습니다

플러그인 메인 클라이언트를 중심으로 backend, relation runtime, recommendation/action payload를 다시 묶었습니다. 사용자는 Obsidian 안에서 현재 note 기준 검색, 근거 확인, 다음 작업 제안까지 끊기지 않는 흐름을 보도록 설계했습니다.

내 역할

FastAPI backend, plugin main client, relation runtime, recommendation/action payload를 직접 설계하고 구현했습니다. 실험용 UI는 Streamlit Ops로 분리해 운영 화면과 실제 사용자 화면의 역할도 나눴습니다.

핵심 설계 1

note 본문, linked note, typed relation schema를 retrieval 기준으로 결합해 relation-aware retrieval 흐름을 만들었습니다. 단순 유사도 검색이 아니라 현재 문서 주변 맥락을 함께 읽는 구조로 바꾸는 것이 핵심이었습니다.

핵심 설계 2

검색 결과를 source list에서 끝내지 않고 recommendation kind, action title, relation path text를 담는 action-ready payload로 확장했습니다. 덕분에 답변 이후의 후속 행동까지 인터페이스 안에서 바로 제안할 수 있게 됐습니다.

선택 이유

RAG는 답변만 좋아져도 실제 사용성이 좋아진다고 보기 어렵습니다. 그래서 source, recommendation, next action을 같은 화면에 두고 사용자가 다시 탐색을 시작하지 않아도 되게 만들었습니다.

채용 포인트

기능을 더한 것이 아니라, 문서 문맥과 후속 행동이 이어지는 전체 사용자 흐름을 기준으로 구조를 다시 설계했습니다. 즉 AI 기능 구현보다 제품 경험 재설계 능력이 더 선명하게 드러나는 장입니다.

Obsidian RAG 03 Outcome

답변형 도구가 아니라 실행 중심의 지식 워크스페이스로 정리했습니다

결과적으로 이 프로젝트는 챗봇 데모보다 실행 중심의 로컬 지식 워크스페이스로 설명하는 편이 더 정확해졌습니다. 현재 문서와 연결 문서, 추천 액션이 한 흐름으로 이어지면서 RAG의 가치가 작업 경험으로 보이기 시작했습니다.

결과

검색 결과를 읽고 끝나는 구조가 아니라, 후속 문서 탐색과 다음 작업까지 이어지는 흐름을 만들었습니다. 질문 한 번이 답변 한 개로 끝나지 않고 실제 작업 전환의 출발점이 되도록 정리했습니다.

설명 가능성

왜 이 문서가 추천됐는지, 어떤 relation path를 탔는지, 다음 행동이 왜 나왔는지를 화면에서 바로 설명할 수 있습니다. 그래서 성능 수치가 없어도 아키텍처와 UX를 함께 설득하기 좋습니다.

제품 관점

Obsidian 안에서 실제로 쓰는 로컬 지식 워크플로로 정리되며, 도구가 아니라 제품형 워크스페이스 경험으로 보이게 됐습니다. 포트폴리오에서도 문서 AI를 작업 도구로 바꾼 사례로 소개할 수 있게 됐습니다.

배운 점

RAG는 retrieval 품질만 올려서는 충분하지 않았습니다. schema 설계, relation runtime, action 설계가 함께 붙어야 실제 사용성이 생긴다는 점을 명확하게 확인했습니다.

채용 포인트

RAG 성능 튜닝만 한 프로젝트가 아니라, 실제 사용 흐름을 설계한 로컬 지식 워크스페이스 사례로 소개할 수 있습니다. 채용관에게는 기술 적용보다 사용 구조를 설계하는 역량이 남는 카드입니다.

Generative Pipeline 01 Problem

부분 수정이 생겨도 전체를 다시 돌려야 해서 비용과 실패 리스크가 컸습니다

상세페이지 생성은 한 번 예쁘게 만드는 것보다, 수정 요청이 들어왔을 때 얼마나 안정적으로 다시 만들 수 있는지가 더 중요했습니다. 그런데 기존 흐름은 일부 수정에도 전체 생성 파이프라인을 다시 태워야 해서 비용과 실패 리스크가 컸습니다.

배경

상세페이지 제작은 기획, 배경 생성, 후처리, 문구 배치, 수정 반복이 계속 이어지는 작업입니다. 그래서 일회성 이미지 생성 모델 데모만으로는 실제 제작 흐름을 설명하기 어려웠습니다.

실제 불편

배경만 바꾸고 싶거나 카피만 수정하고 싶은 상황이 자주 생겼는데, 어디부터 다시 생성해야 하는지 경계가 불명확했습니다. 수정 포인트가 늘어날수록 전체 비용과 대기 시간이 더 커졌습니다.

왜 어려웠나

생성 단계 하나가 실패하면 전체 결과물을 다시 확인해야 했고, 어떤 단계가 병목인지도 설명하기 어려웠습니다. 운영 관점에서 보면 품질보다 복구 가능성이 더 큰 문제였습니다.

해결 목표

예쁜 결과물 몇 장보다, 부분 수정과 단계별 재시도를 버틸 수 있는 제작 파이프라인을 만드는 것이 목표였습니다. 생성 AI를 결과물 생성기가 아니라 운영 가능한 제작 시스템으로 바꾸고 싶었습니다.

채용 포인트

생성 AI를 결과물 데모가 아니라, 반복 수정과 실패 리스크를 다루는 운영 구조 문제로 재정의한 사례입니다. 실무형 시스템 감각이 가장 먼저 드러나는 문제 정의 카드입니다.

Generative Pipeline 02 Approach

Director와 5-stage 파이프라인으로 책임 분리와 복구 경로를 만들었습니다

그래서 Director 중심 오케스트레이션과 5-stage 파이프라인으로 책임을 나눴습니다. 기획과 실행을 분리하고, failover와 page regenerate를 넣어 수정 가능한 서비스형 생성 구조로 다시 만들었습니다.

내 역할

FastAPI orchestration, Director, 5-stage image pipeline, LLM failover, regenerate 경로를 직접 설계했습니다. 단순 모델 호출이 아니라 상태를 다루는 시스템 설계 쪽에 무게를 두었습니다.

핵심 설계 1

기획 단계에서 manifest를 만들고, 실행 단계에서 각 페이지를 처리하도록 초기화와 실행을 분리했습니다. 이 구조 덕분에 어떤 산출물이 왜 나왔는지, 무엇을 다시 돌려야 하는지 설명이 쉬워졌습니다.

핵심 설계 2

segmenter, background generation, upscaler, color/post-process, ad composition으로 단계를 끊고 각 단계의 시간과 실패 지점을 추적했습니다. 부분 regenerate와 재시도도 페이지 단위로 처리할 수 있게 했습니다.

선택 이유

실무에서는 전체 재생성보다 수정 요청 대응과 실패 복구가 더 자주 일어납니다. 그래서 모델 성능을 더 붙이는 것보다, 책임 분리와 복구 경로를 먼저 설계하는 게 우선이라고 판단했습니다.

채용 포인트

모델 조합보다 먼저 시스템 책임을 나누고 재생성 경로를 설계했다는 점이, 서비스형 AI를 다루는 역량으로 연결됩니다. 생성형 AI를 운영 가능한 파이프라인으로 바꾸는 사고가 드러나는 카드입니다.

Generative Pipeline 03 Outcome

결과 이미지보다 운영 가능한 생성 시스템 설계 경험으로 정리했습니다

결과적으로 이 프로젝트는 이미지 생성 데모보다 운영 가능한 제작 시스템 설계 경험으로 보는 편이 더 맞아졌습니다. 산출물 품질뿐 아니라, 수정 요청과 실패 이후를 다루는 구조가 함께 생겼기 때문입니다.

결과

페이지 단위 regenerate와 부분 수정이 가능한 흐름을 만들면서 제작 프로세스를 다시 설명할 수 있게 됐습니다. 특히 무엇을 다시 돌려야 하는가가 구조적으로 명확해졌습니다.

설명 가능성

병목과 실패 지점을 단계별로 추적하고 공유할 수 있어, 결과물뿐 아니라 생성 과정도 함께 설명 가능합니다. 발표나 리뷰에서도 기술 선택 이유를 자연스럽게 연결할 수 있게 됐습니다.

서비스 관점

결과 이미지 몇 장이 아니라, 생성 이후 수정과 운영까지 포함한 전체 시스템이 포트폴리오의 핵심이 됐습니다. 즉 AI를 사용하는 사람이 아니라 AI 워크플로를 설계하는 사람으로 보이게 됩니다.

배운 점

생성 AI도 결국 운영 시스템이기 때문에 오케스트레이션, 상태 관리, 실패 대응이 품질만큼 중요했습니다. 좋은 모델 조합만으로는 실무 구조가 만들어지지 않는다는 점을 분명히 배웠습니다.

채용 포인트

생성 AI를 사용한 경험이 아니라, 생성 이후 수정과 복구를 포함한 전체 서비스 파이프라인을 설계한 경험으로 보여줄 수 있습니다. 채용관에게는 운영 가능한 AI 시스템 경험으로 읽히는 마무리 카드입니다.

CFD Trading 01 Problem

데이터, 판단 근거, 운영 화면이 분리되면 설명과 검증이 어려웠습니다

CFD 프로젝트에서는 예측 정확도 하나보다, 데이터를 어떤 상태로 해석했고 그 판단이 운영 화면으로 어떻게 이어지는지가 더 중요했습니다. 그런데 데이터, 판단 로직, 운영 UI가 분리되면 실제 의사결정 근거를 설명하거나 검증하기가 매우 어려웠습니다.

배경

트레이딩에서는 숫자 하나보다 현재 시장 상태를 어떻게 읽고 있는지가 더 중요합니다. 운영자가 같은 데이터를 다시 보고 판단 근거를 설명할 수 있어야 시스템을 믿고 다룰 수 있습니다.

실제 불편

시장 데이터 수집, 규칙 판단, ML 보조 신호, 운영 화면이 따로 움직이면 분석 결과가 어떤 행동으로 이어졌는지 추적하기 어렵습니다. 특히 시연이나 검증 상황에서 설명이 한 번에 끊깁니다.

왜 어려웠나

규칙 기반 판단과 ML 보조 판단의 역할이 다르고, API와 대시보드도 따로 설계되면 공통 언어가 사라집니다. 그러면 결과는 있어도 왜 그런 판단이 나왔는지 설명 가능성이 약해집니다.

해결 목표

예측 모델 하나를 자랑하는 것이 아니라, 데이터 해석부터 운영 검증까지 같은 언어로 이어지는 설명 가능한 분석 시스템을 만드는 것이 목표였습니다. 분석과 운영을 따로 두지 않는 구조를 만드는 것이 핵심이었습니다.

채용 포인트

예측 정확도 중심이 아니라, 데이터 해석부터 운영 검증까지 이어지는 설명 가능한 시스템 문제로 접근했습니다. 머신러닝을 단독 모델이 아니라 시스템 맥락에서 다루는 관점이 드러나는 카드입니다.

CFD Trading 02 Approach

정규화 파이프라인과 상태 해석 레이어를 중심으로 흐름을 다시 묶었습니다

그래서 Market·MT5 데이터를 먼저 정규화하고, 그 위에 상태 해석 레이어를 세운 뒤 규칙 판단, ML assist, FastAPI, Next.js dashboard를 한 흐름으로 묶었습니다. 각 레이어가 같은 상태 정보를 보도록 맞춘 것이 핵심이었습니다.

내 역할

데이터 처리 파이프라인, state interpretation, ML assist 흐름, FastAPI runtime API, Next.js dashboard를 직접 연결했습니다. 분석 코드, 운영 API, 화면을 한 사람의 시선으로 일관되게 묶는 데 집중했습니다.

핵심 설계 1

수집한 데이터를 바로 쓰지 않고 정규화와 상태 해석을 거친 공통 포맷으로 만들었습니다. 덕분에 분석, 판단, 대시보드가 서로 다른 숫자를 말하지 않고 같은 기준을 공유하게 됐습니다.

핵심 설계 2

규칙 기반 판단 위에 ML assist를 보조 레이어로 붙이고, runtime·trades·ops API와 dashboard가 같은 상태와 이벤트를 읽도록 연결했습니다. 설명 가능한 기본 흐름을 유지하면서도 보조 예측을 붙일 수 있게 했습니다.

선택 이유

운영 단계에서는 모델 성능 수치보다 지금 어떤 상태인지 바로 보이는 것이 더 중요합니다. 그래서 API와 대시보드를 별도 결과물이 아니라 같은 판단 흐름의 일부로 설계했습니다.

채용 포인트

데이터 엔지니어링, 해석 로직, 운영 API, 대시보드를 따로 만들지 않고 하나의 설명 가능한 시스템으로 연결한 사례입니다. 채용관이 보기에도 end-to-end 감각이 분명하게 드러나는 장입니다.

CFD Trading 03 Outcome

예측 모델보다 설명 가능한 트레이딩 분석 시스템으로 정리했습니다

결과적으로 이 프로젝트는 예측 모델 데모보다 설명 가능한 트레이딩 분석 시스템으로 소개하는 편이 훨씬 설득력 있어졌습니다. 같은 데이터가 분석, 판단, API, 대시보드까지 일관되게 이어지는 구조가 만들어졌기 때문입니다.

결과

분석 결과가 문서나 코드에만 머무르지 않고 운영 화면에서 바로 확인되는 구조를 만들었습니다. 운영자는 상태, 판단 근거, 최근 이벤트를 같은 맥락에서 읽을 수 있게 됐습니다.

설명 가능성

시장 상태, 규칙 판단, ML 보조 판단이 어떻게 결합되는지 한 흐름에서 보여줄 수 있어 시연과 검증이 쉬워졌습니다. 포트폴리오에서도 왜 이런 구조를 만들었는가를 설명하기 수월해졌습니다.

운영 관점

dashboard, API, 테스트, 문서를 함께 두어 시스템 변화가 운영에 어떤 영향을 주는지 빠르게 확인할 수 있습니다. 그래서 단발성 분석 프로젝트보다 유지 가능한 운영형 프로젝트로 읽히게 됐습니다.

배운 점

데이터 프로젝트는 모델링만 잘한다고 끝나지 않았습니다. 설명 가능성, 운영 확인성, 문서화가 같이 있어야 구조가 오래 유지된다는 점을 분명히 체감했습니다.

채용 포인트

데이터 분석과 모델링을 넘어서, 판단 근거를 설명 가능한 운영 구조로 만드는 역량을 보여주는 사례가 됩니다. 머신러닝을 실제 시스템으로 연결할 수 있는 사람이라는 인상을 남기는 마무리 카드입니다.
