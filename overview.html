Obsidian RAG 01 Problem

기록은 쌓이지만 구조와 다음 행동은 자동으로 생기지 않았습니다

Obsidian에는 기록이 계속 쌓이지만, 그 기록이 자동으로 살아 있는 지식망이 되거나 다음 작업으로 이어지지는 않았습니다. 그래서 이 프로젝트는 챗봇 하나를 더 만드는 일이 아니라, 현재 노트 기준의 구조와 후속 행동을 복구하는 문제로 다시 정의했습니다.

문제 정의

V1은 Streamlit 중심 실험 환경에 가까워 retrieval 품질을 확인하기에는 좋았지만, 실제 Obsidian 작업 문맥 안에서 바로 쓰기에는 동선이 멀었습니다. 사용자는 노트를 읽다가도 별도 콘솔로 이동해 질문하고 다시 돌아와야 했습니다.

왜 일반 RAG로는 부족했나

일반적인 RAG는 비슷한 문서를 찾아 답변을 만들 수는 있어도, 지금 읽는 노트를 기준으로 어떤 구현 문서가 이어지고 어떤 리뷰가 연결되며 무엇을 바로 실행해야 하는지는 설명하기 어려웠습니다.

내가 다시 잡은 기준

핵심은 답변 정확도만 올리는 것이 아니라, 현재 문서를 기준으로 근거 문서, 추천 문서, 다음 액션이 같은 화면에서 이어지는 작업형 RAG를 만드는 것이었습니다. 검색보다도 읽기에서 실행으로 이어지는 UX를 복구하는 것이 더 중요한 문제라고 판단했습니다.

구현 강점

문제를 단순한 챗봇 성능 이슈가 아니라, 실제 사용 흐름과 UX가 끊기는 구조 문제로 다시 정의한 사례입니다. 기능 추가보다 문제를 다시 세우는 기준이 먼저 보이는 시작 장입니다.

이 프로젝트의 구현 강점

RAG를 성능 실험으로만 보지 않고, 실제 작업 흐름이 왜 끊기는지부터 다시 해석했다는 점이 중요합니다. 그래서 이 장은 기술 스택보다 문제를 다시 정의하는 시선이 먼저 남도록 구성했습니다.

Obsidian RAG 02 Approach

문서를 역할과 관계를 가진 노드로 다시 해석하고, 그 구조를 여러 기능이 같이 쓰게 만들었습니다

이 프로젝트의 핵심은 relation을 retrieval 보정용 부가 기능으로 두지 않고, recommendation, action, graph UI까지 함께 쓰는 공통 인프라로 올린 점입니다. 사용자는 Obsidian 안에서 현재 노트 기준 검색, 근거 확인, 다음 작업 제안까지 끊기지 않는 흐름을 보게 됩니다.

Structured-first

`note_type_auto` 와 `doc_role_auto` 를 분리해 문서 형식과 사고 흐름상 역할을 함께 해석했습니다. 문서를 단순 청크가 아니라 역할을 가진 노드로 보는 것이 출발점이었습니다.

Typed relation

`implements`, `review_of`, `next_action_for`, `decision_for`, `follow_up` 같은 설명 가능한 relation을 만들고, explicit wikilink, related file, semantic tag를 함께 사용해 relation 품질을 올렸습니다.

Relation runtime

Phase 2에서는 relation metadata를 실제 런타임 관계망으로 승격해 1-hop / 2-hop traversal, relation path scoring, relation path explanation을 공통 인프라로 만들었습니다.

Action-ready payload

추천을 source list에서 끝내지 않고 `recommendation_kind`, `action_title`, `relation_path_text`, `hop_count` 를 담는 payload로 확장했습니다. 그래서 recommendation이 바로 action engine과 graph UI의 입력이 됩니다.

이 프로젝트의 구현 강점

기능을 더한 것이 아니라, taxonomy, relation schema, runtime, plugin UI를 하나의 설계 언어로 묶었습니다. 이 장에서는 AI 기능 구현보다 시스템 설계 능력이 더 또렷하게 드러납니다.

Obsidian RAG 03 Outcome

검색 결과를 추천, 액션, 그래프 설명으로 연결한 in-note workspace로 정리했습니다

핵심은 답변 하나를 더 잘 만드는 것이 아니라, relation path를 source, recommendation, action, graph가 같이 공유하도록 만들어 왜 이 노트가 나왔는지와 다음에 무엇을 할지를 같은 화면에서 설명하게 한 점입니다. 그래서 이 프로젝트는 챗봇보다 실행 중심 지식 워크스페이스로 소개하는 편이 더 정확해졌습니다.

무엇이 달라졌나

V1에서는 answer 중심이었던 흐름이 V2에서 in-note workspace로 바뀌었습니다. 현재 노트 기준 검색, follow-up note, suggested action, workflow tool이 같은 제품 안에서 이어지기 시작했습니다.

검증 상태

relation graph runtime, recommendation payload, action engine을 acceptance 기준으로 검증했고, 회귀 테스트와 build를 같이 묶어 아이디어 수준이 아니라 실제로 유지되는 흐름임을 확인했습니다.

왜 설득력 있나

RAG 성능 튜닝만 이야기하는 대신, relation path를 recommendation과 action으로 연결한 구조를 보여주기 때문에 처음 보는 사람도 "왜 일반 RAG와 다른가"를 빠르게 이해할 수 있습니다.

제품 의미

결과적으로 Obsidian RAG는 로컬 문서를 잘 찾는 챗봇이 아니라, 기록을 구조화하고 관계 기반으로 재사용하는 개인 지식 워크스페이스로 정리됐습니다. 이 점이 포트폴리오 설명의 핵심 메시지입니다.

이 프로젝트의 구현 강점

문서 AI를 검색 데모로 끝내지 않고, 구조화, relation runtime, action UX, 검증까지 묶은 제품형 사례입니다. 그래서 AI 응용보다 시스템을 제품으로 정리하는 역량이 더 크게 남는 마무리 장이 됩니다.

Generative Pipeline 01 Problem

부분 수정이 생겨도 전체를 다시 돌려야 해서 비용과 실패 리스크가 컸습니다

상세페이지 생성은 한 번 예쁘게 만드는 것보다, 수정 요청이 들어왔을 때 얼마나 안정적으로 다시 만들 수 있는지가 더 중요했습니다. 그런데 기존 흐름은 일부 수정에도 전체 생성 파이프라인을 다시 태워야 해서 비용과 실패 리스크가 컸습니다.

배경

상세페이지 제작은 기획, 배경 생성, 후처리, 문구 배치, 수정 반복이 계속 이어지는 작업입니다. 그래서 일회성 이미지 생성 모델 데모만으로는 실제 제작 흐름을 설명하기 어려웠습니다.

실제 불편

배경만 바꾸고 싶거나 카피만 수정하고 싶은 상황이 자주 생겼는데, 어디부터 다시 생성해야 하는지 경계가 불명확했습니다. 수정 포인트가 늘어날수록 전체 비용과 대기 시간이 더 커졌습니다.

왜 어려웠나

생성 단계 하나가 실패하면 전체 결과물을 다시 확인해야 했고, 어떤 단계가 병목인지도 설명하기 어려웠습니다. 운영 관점에서 보면 품질보다 복구 가능성이 더 큰 문제였습니다.

해결 목표

예쁜 결과물 몇 장보다, 부분 수정과 단계별 재시도를 버틸 수 있는 제작 파이프라인을 만드는 것이 목표였습니다. 생성 AI를 결과물 생성기가 아니라 운영 가능한 제작 시스템으로 바꾸고 싶었습니다.

이 프로젝트의 구현 강점

생성 AI를 결과물 데모가 아니라, 반복 수정과 실패 리스크를 다루는 운영 구조 문제로 재정의한 사례입니다. 실무형 시스템 감각이 가장 먼저 드러나는 문제 정의 카드입니다.

Generative Pipeline 02 Approach

Director와 5-stage 파이프라인으로 책임 분리와 복구 경로를 만들었습니다

그래서 Director 중심 오케스트레이션과 5-stage 파이프라인으로 책임을 나눴습니다. 기획과 실행을 분리하고, failover와 page regenerate를 넣어 수정 가능한 서비스형 생성 구조로 다시 만들었습니다.

내 역할

FastAPI orchestration, Director, 5-stage image pipeline, LLM failover, regenerate 경로를 직접 설계했습니다. 단순 모델 호출이 아니라 상태를 다루는 시스템 설계 쪽에 무게를 두었습니다.

핵심 설계 1

기획 단계에서 manifest를 만들고, 실행 단계에서 각 페이지를 처리하도록 초기화와 실행을 분리했습니다. 이 구조 덕분에 어떤 산출물이 왜 나왔는지, 무엇을 다시 돌려야 하는지 설명이 쉬워졌습니다.

핵심 설계 2

segmenter, background generation, upscaler, color/post-process, ad composition으로 단계를 끊고 각 단계의 시간과 실패 지점을 추적했습니다. 부분 regenerate와 재시도도 페이지 단위로 처리할 수 있게 했습니다.

선택 이유

실무에서는 전체 재생성보다 수정 요청 대응과 실패 복구가 더 자주 일어납니다. 그래서 모델 성능을 더 붙이는 것보다, 책임 분리와 복구 경로를 먼저 설계하는 게 우선이라고 판단했습니다.

이 프로젝트의 구현 강점

모델 조합보다 먼저 시스템 책임을 나누고 재생성 경로를 설계했다는 점이, 서비스형 AI를 다루는 역량으로 연결됩니다. 생성형 AI를 운영 가능한 파이프라인으로 바꾸는 사고가 드러나는 카드입니다.

Generative Pipeline 03 Outcome

결과 이미지보다 운영 가능한 생성 시스템 설계 경험으로 정리했습니다

결과적으로 이 프로젝트는 이미지 생성 데모보다 운영 가능한 제작 시스템 설계 경험으로 보는 편이 더 맞아졌습니다. 산출물 품질뿐 아니라, 수정 요청과 실패 이후를 다루는 구조가 함께 생겼기 때문입니다.

결과

페이지 단위 regenerate와 부분 수정이 가능한 흐름을 만들면서 제작 프로세스를 다시 설명할 수 있게 됐습니다. 특히 무엇을 다시 돌려야 하는가가 구조적으로 명확해졌습니다.

설명 가능성

병목과 실패 지점을 단계별로 추적하고 공유할 수 있어, 결과물뿐 아니라 생성 과정도 함께 설명 가능합니다. 발표나 리뷰에서도 기술 선택 이유를 자연스럽게 연결할 수 있게 됐습니다.

서비스 관점

결과 이미지 몇 장이 아니라, 생성 이후 수정과 운영까지 포함한 전체 시스템이 포트폴리오의 핵심이 됐습니다. 즉 AI를 사용하는 사람이 아니라 AI 워크플로를 설계하는 사람으로 보이게 됩니다.

배운 점

생성 AI도 결국 운영 시스템이기 때문에 오케스트레이션, 상태 관리, 실패 대응이 품질만큼 중요했습니다. 좋은 모델 조합만으로는 실무 구조가 만들어지지 않는다는 점을 분명히 배웠습니다.

이 프로젝트의 구현 강점

생성 AI를 사용한 경험이 아니라, 생성 이후 수정과 복구를 포함한 전체 서비스 파이프라인을 설계한 경험으로 보여줄 수 있습니다. 운영 가능한 AI 시스템 경험으로 읽히는 마무리 카드입니다.

CFD Trading 01 Problem

차트는 보이는데 판단은 남지 않았고, 결과는 남아도 과정은 남지 않았습니다

이 프로젝트는 "수익률이 좋은 전략 하나"보다, 차트의 위치와 반응을 사람이 어떻게 해석했는지, 왜 들어갔고 왜 기다렸고 왜 잘랐는지를 데이터 구조로 남기는 문제에서 시작했습니다. 점수 하나로는 시장을 설명하기 어려웠고, 차트 직관이 데이터로 남지 않으면 학습과 검증도 반복되지 않았습니다.

점수만으로는 부족했다

가격이 어디에 있는지, 그 위치에서 어떤 반응이 나왔는지, continuation인지 reversal인지 같은 흐름을 raw score 하나로 설명하는 데는 한계가 있었습니다.

차트 직관을 남겨야 했다

사람은 1분봉만 보고도 조용한 장, 추세 지속장, 브레이크아웃 직전, 페이크아웃 반전 같은 장세를 읽습니다. 하지만 이 직관이 데이터로 남지 않으면 나중에 학습과 검증을 이어가기 어렵습니다.

결과만으론 부족했다

손익만 남으면 회고는 가능해도 학습은 어렵습니다. 왜 들어갔고 왜 기다렸고 왜 잘랐는지가 lifecycle로 남아야 다음 실험과 검증이 가능합니다.

다시 잡은 목표

핵심 목표는 자동매매 룰을 더 붙이는 것이 아니라, 시장 해석과 거래 lifecycle을 설명 가능한 state 구조로 올리고 다시 compact dataset으로 남기는 것이었습니다.

이 프로젝트의 구현 강점

트레이딩을 수익률 데모가 아니라 해석 구조와 기록 구조의 문제로 다시 정의했다는 점이 중요합니다. 모델링보다 문제를 시스템 관점에서 재정의하는 시선이 먼저 남는 카드입니다.

CFD Trading 02 Approach

position -> response -> state -> forecast -> decision lifecycle 위에 teacher-state 25를 얹었습니다

이 프로젝트의 핵심은 BB 점수 몇 점 같은 납작한 설명이 아니라, 이미 있는 엔진을 사람이 이해하는 state와 학습 가능한 데이터로 재정렬한 점입니다. raw score는 남아 있어도 메인 판단축이 아니고, 메인 판단은 위치와 반응, state와 forecast의 흐름에 가깝습니다.

기획

`teacher-state 25`, `micro-structure Top10`, `compact schema`, `labeling QA`, `experiment tuning`을 먼저 고정했습니다. 새 엔진을 처음부터 만드는 대신 기존 `position / state / forecast` 구조를 최대한 활용했습니다.

생성

`body_size_pct_20`, `wick ratio`, `range compression`, `volume burst/decay`, `gap fill progress` 등 micro-structure Top10을 넣고, OHLCV helper -> StateRawSnapshot -> forecast harvest -> compact bridge로 이어지는 경로를 만들었습니다.

수정

`payload flatness`, `entry_atr_ratio_flat`, confusion pair, execution handoff gate, runtime drift를 보정 대상으로 두고 지속적으로 튜닝했습니다. "만들었다"보다 "붙여보고 수정했다"가 더 중요한 구조였습니다.

핵심 결과물

결과적으로 이 프로젝트는 시장 해석, 거래 lifecycle 기록, 학습 가능한 compact dataset 생성을 동시에 다루는 구조가 됐습니다. 룰을 더 추가한 것이 아니라 엔진을 state와 dataset 중심으로 다시 정렬한 셈입니다.

이 프로젝트의 구현 강점

지표 하나를 더 만든 것이 아니라, 시장 해석을 lifecycle과 teacher-label 구조로 올리는 시스템 설계를 했다는 점이 핵심입니다. end-to-end 데이터 제품 감각이 분명하게 드러나는 장입니다.

CFD Trading 03 Outcome

시장 해석과 거래 lifecycle을 teacher-label 데이터로 남기는 시스템으로 정리했습니다

결과적으로 이 프로젝트는 예측 모델 데모보다, 시장 상태를 점점 더 넓게 읽고 기록하는 시스템으로 설명하는 편이 더 설득력 있어졌습니다. 지금 단계의 핵심은 coverage를 늘리면서 실제 labeled row와 QA 흐름을 현실화하는 데 있습니다.

현재 상태

`micro-structure Top10`, `teacher_pattern_* schema`, `state25 labeler`, `Step 8 QA gate`, `bounded + richer backfill`까지 구축됐고, 현재는 `E5` 재확인과 장기 runtime 관찰 단계입니다.

지금 남은 경고

`unlabeled_rows_present`, `rare_pattern_watch_triggered`, `low_confidence_review_required` 는 아직 남아 있습니다. 이 점을 숨기지 않고 coverage와 support를 넓히는 단계라고 설명하는 편이 오히려 더 설득력 있습니다.

왜 설득력 있나

이 프로젝트는 수익률 그래프 하나보다, 상태를 어떻게 읽고 어떤 거래 lifecycle이 compact row로 남는지 보여주기 때문에 처음 보는 사람도 "왜 이게 데이터 프로젝트인가"를 빠르게 이해할 수 있습니다.

다음 단계

앞으로는 coverage, activation funnel, wait/exit quality mix를 운영 화면으로 더 강하게 시각화해 `CFD Lifecycle Intelligence Board` 형태로 묶는 것이 가장 자연스러운 다음 단계입니다.

이 프로젝트의 구현 강점

트레이딩을 결과 맞추기보다 판단 과정을 기록하고 검증 가능한 데이터 구조로 바꾼 사례입니다. 머신러닝을 시스템과 데이터 제품의 관점에서 다루는 사람이라는 인상을 남기는 마무리 장입니다.

How I Analyze Data

데이터를 이렇게 분석하고 정리합니다

ML Workflow

모델보다 먼저 데이터 단위, 지표, 실패 케이스를 정리합니다

회귀, 분류, 문서형 RAG를 같은 방식으로 다루지 않습니다. 먼저 예측 대상과 입력 표현을 정의하고, EDA로 변수와 구조를 확인한 뒤, 문제에 맞는 지표와 오류 패턴을 기준으로 다음 실험을 설계합니다. 그 과정은 노트와 포트폴리오에 다시 정리해 다음 프로젝트의 판단 근거로 이어갑니다.

EDA -> Feature -> Metric Raw + Summary + Metadata Error Note -> Next Experiment
질문과 지표를 먼저 정의합니다

문제 유형에 맞는 기준이 먼저 정해져야 실험이 흔들리지 않습니다. 회귀에서는 RMSLE 같은 오차 기준을, 불균형 분류에서는 Recall, F1, ROC-AUC처럼 놓치면 안 되는 실패 패턴을 먼저 보고 baseline과 feature를 조정합니다.
