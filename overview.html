Obsidian RAG Story

현재 노트 문맥 안에서 바로 답을 찾고 다음 액션까지 이어지게 만들고 싶었습니다

검색 결과만 보여주는 RAG가 아니라, 현재 문서와 연결된 관계와 후속 행동까지 Obsidian 안에서 이어지는 워크스페이스를 목표로 했습니다.

01 Problem

노트는 쌓였는데 현재 문서 기준으로 무엇을 읽고 무엇을 할지 바로 보이지 않았습니다

단순 검색은 조각난 결과를 잘 찾지만, 지금 보고 있는 노트와 연결된 맥락과 우선순위를 한 번에 보여주기에는 부족했습니다.

  • 현재 노트와 관련 노트 사이의 관계를 같이 읽어야 했습니다.
  • 답변 뒤에 바로 열어볼 문서와 실행 액션 제안이 필요했습니다.

02 Approach

현재 노트 문맥, relation-aware retrieval, 추천 액션 흐름을 한 화면으로 묶었습니다

플러그인 메인 클라이언트와 Streamlit Ops를 분리하고, relation runtime으로 현재 문서 주변의 연결 노트를 함께 가져오도록 설계했습니다.

  • 현재 note, linked note, relation schema를 retrieval 기준으로 결합했습니다.
  • 답변, source, recommendation, next action을 같은 흐름으로 보여주게 했습니다.

03 Outcome

검색형 도구보다 작업형 지식 워크스페이스에 가까운 구조로 정리했습니다

결과적으로 이 프로젝트는 질의응답 성능만이 아니라, 문서 탐색과 후속 행동을 같은 작업 맥락 안에 두는 설계 경험을 보여줍니다.

  • 답변과 근거 문서, 추천 문서, 액션 흐름이 함께 보입니다.
  • Obsidian 안에서 바로 쓰는 로컬 지식 워크플로 형태로 설명할 수 있습니다.

Generative Pipeline Story

한 번 실패하면 전체를 다시 돌려야 하는 원샷 생성 흐름을 서비스형 파이프라인으로 바꾸고 싶었습니다

상세페이지 제작은 부분 수정이 잦기 때문에, 기획과 생성을 분리하고 단계별 실패를 다룰 수 있는 구조가 더 중요했습니다.

01 Problem

기획, 생성, 편집이 분리되어 있어 수정이 생기면 전체 파이프라인 비용이 너무 컸습니다

광고 상세페이지는 한 장만 바꾸고 싶은 경우가 많지만, 기존 흐름은 일부 수정에도 전체 생성을 다시 돌려야 했습니다.

  • 단계 하나가 실패하면 전체 작업이 멈추는 구조였습니다.
  • 부분 수정과 재생성이 어려워 실무형 제작 흐름과 거리가 있었습니다.

02 Approach

Director 중심 오케스트레이션과 5-stage 파이프라인으로 단계별 책임을 나눴습니다

입력 정보로부터 기획, 배경 생성, 업스케일, 보정, 광고 조합까지를 분리하고 상태 추적과 failover 경로를 설계했습니다.

  • AIDA 기반 기획 흐름과 stage별 재실행 포인트를 만들었습니다.
  • OpenAI, Gemini failover와 페이지 단위 regenerate 경로를 넣었습니다.

03 Outcome

결과물 몇 장보다도 수정 가능하고 운영 가능한 생성 시스템이라는 점이 핵심이 됐습니다

이 프로젝트는 단일 모델 결과보다, 생성 이후의 수정과 재시도를 포함한 실제 제작 파이프라인을 설계한 경험으로 정리됩니다.

  • 병목과 실패 지점을 단계별로 기록하고 설명할 수 있습니다.
  • 부분 수정과 재생성이 가능해 실무형 생성 워크플로에 더 가까워졌습니다.

CFD Trading Story

데이터, 판단 로직, 운영 화면이 따로 노는 문제를 설명 가능한 분석 흐름으로 묶고 싶었습니다

시장 데이터를 봤다는 사실보다, 그 데이터를 어떻게 해석하고 어떤 상태로 API와 대시보드까지 연결했는지가 더 중요하다고 봤습니다.

01 Problem

시장 데이터와 판단 근거, 운영 화면이 분리되면 검증과 설명이 어려워집니다

데이터 수집은 따로, 판단 로직은 따로, 운영 확인은 따로 움직이면 실제로 무엇을 근거로 의사결정했는지 추적하기가 힘들어집니다.

  • 규칙 기반 판단과 ML 결과가 분리되면 해석이 어렵습니다.
  • API, 대시보드, 문서가 연결되지 않으면 유지보수도 약해집니다.

02 Approach

정규화 파이프라인과 상태 해석 레이어를 기준으로 전체 시스템을 다시 묶었습니다

Market, MT5 데이터를 정규화하고 상태와 신호 해석 레이어를 둔 뒤, 규칙 기반 판단과 ML 보조 판단을 운영 API로 연결했습니다.

  • state interpretation을 중심으로 feature pipeline과 runtime API를 묶었습니다.
  • FastAPI 운영 API와 Next.js 대시보드가 같은 상태 정보를 보도록 구성했습니다.

03 Outcome

예측 모델 하나보다 데이터부터 운영 화면까지 이어지는 분석 시스템으로 설명할 수 있게 됐습니다

포트폴리오 관점에서도 이 프로젝트는 트레이딩 모델보다, 데이터 처리와 상태 해석을 운영 가능한 시스템으로 연결한 경험을 보여줍니다.

  • 같은 데이터를 분석, 판단, API, 대시보드에서 재사용할 수 있습니다.
  • 시장 상태와 판단 근거를 실제 운영 화면에서 함께 확인할 수 있습니다.

ML Workflow

모델보다 먼저 데이터 단위, 지표, 실패 케이스를 정리합니다

회귀, 분류, 문서형 RAG를 같은 방식으로 다루지 않습니다. 먼저 예측 대상과 입력 표현을 정의하고, EDA로 변수와 구조를 확인한 뒤, 문제에 맞는 지표와 오류 패턴을 기준으로 다음 실험을 설계합니다. 그 과정은 노트와 포트폴리오에 다시 정리해 다음 프로젝트의 판단 근거로 이어갑니다.

EDA -> Feature -> Metric Raw + Summary + Metadata Error Note -> Next Experiment
질문과 지표를 먼저 정의합니다

문제 유형에 맞는 기준이 먼저 정해져야 실험이 흔들리지 않습니다. 회귀에서는 RMSLE 같은 오차 기준을, 불균형 분류에서는 Recall, F1, ROC-AUC처럼 놓치면 안 되는 실패 패턴을 먼저 보고 baseline과 feature를 조정합니다.
