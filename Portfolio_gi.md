

#  포트폴리오 작성 가이드라인 v3 (깃블로그용)

---

너는 내 코드베이스를 기반으로 **깃블로그에 올릴 포트폴리오 문서**를 작성해야 한다.

---

## 0. 이 문서의 역할

이 문서는 GitHub README와 역할이 다르다.

**GitHub README:**
- 이 프로젝트가 뭔지, 기술 스택, 설치법, API 문서, 디렉토리 구조
- 이미 별도로 존재한다. 여기서 다루지 않는다.

**이 문서 (깃블로그):**
- 왜 만들었는지, 뭐가 문제였는지, 왜 이렇게 판단했는지, 결과가 뭔지
- 설계 판단과 사고 과정을 보여주는 문서다.

따라서 이 문서에는 다음을 **쓰지 않는다:**
- 설치 방법, 실행 방법
- API 엔드포인트 목록
- 프로젝트 디렉토리 구조 전체
- 기술 스택 단순 나열 (판단 이유 없이 나열만 하는 것)
- README에 이미 있는 내용의 반복

문서 하단에 GitHub 레포지토리 링크를 달아서 코드와 기술 문서는 그쪽에서 확인하도록 유도한다.

## 1. 문서 작성 원칙

### 톤 & 문체

- **"~했습니다" 체가 아닌 "~했다" 체(건조체)로 쓴다**
- **"~할 수 있다", "~될 수 있다" 같은 가능성 표현을 쓰지 않는다. 실제로 겪은 문제는 "~했다", "~됐다"로 단정한다**
  - ❌ "무관 문서가 섞일 수 있다"
  - ✅ "무관 문서가 섞였다"
  - ❌ "근거가 약해질 수 있다"
  - ✅ "근거가 약해졌다"
- 감상을 쓰지 않는다. "좋은 경험이었다", "많이 배웠다" 같은 문장 금지
- 형용사를 최소화한다. "다양한", "혁신적인", "효과적인" 같은 단어 금지
- 문장은 **한 문장에 하나의 정보**만 담는다
- 두 줄 이상 이어지는 문장은 쪼갠다
- 기술 용어는 정확하게 쓰되, 불필요한 영어 남발은 하지 않는다

### 관점

- **주어는 기술이 아니라 문제다**
  - ❌ "LangChain을 활용하여 RAG를 구현했다"
  - ✅ "문서 검색 정확도가 낮아서 검색 파이프라인을 재설계했다"
- **도구 나열이 아니라 판단 과정을 보여준다**
  - ❌ "ChromaDB를 사용했다"
  - ✅ "로컬에서 빠르게 실험해야 했기 때문에 별도 서버가 필요 없는 ChromaDB를 선택했다"
- **"뭘 했다"가 아니라 "왜 이렇게 했다"를 중심에 둔다**

### 수치 원칙

- 성능, 개선폭, 비용, 속도 등은 반드시 **수치로 표현**한다
- "크게 향상되었다" ❌ → "0.62에서 0.81로 개선되었다" ✅
- "빨라졌다" ❌ → "8초에서 2.8초로 줄었다" ✅
- 정확한 수치를 모르면 코드에서 추정 가능한 근거를 찾아서 쓰고, 추정임을 명시한다
- **config 파라미터 값(threshold, retry 횟수 등)은 "결과"가 아니다. 결과 섹션에 넣지 마라**

## 2. 코드 분석 방법

### 내 코드베이스에서 다음을 파악하라

```
[ ] 이 프로젝트가 해결하려는 문제가 뭔지
    → 입력과 출력을 보면 알 수 있다
    → API 엔드포인트, main 함수, 데이터 흐름을 추적하라

[ ] 전체 파이프라인 흐름이 어떻게 되는지
    → 데이터가 어디서 들어와서 어디로 나가는지
    → 각 단계에서 어떤 처리가 일어나는지
    → 함수 호출 순서를 따라가라

[ ] 어떤 기술/라이브러리를 쓰고 있는지
    → requirements.txt, import문, config 파일 확인
    → 각 기술이 파이프라인의 어느 단계에서 쓰이는지 매핑

[ ] 설계상 특별한 선택이 있는지
    → 단순하게 할 수 있었는데 일부러 복잡하게 한 부분
    → config에서 조정 가능한 파라미터들
    → 주석이나 TODO에 남긴 의도

[ ] 에러 처리, 로깅, 평가 코드가 있는지
    → try/except 패턴
    → 로깅 구조
    → 테스트 코드
    → 평가 스크립트

[ ] 개선의 흔적이 있는지
    → git 히스토리에서 구조 변경
    → 주석 처리된 이전 코드
    → v1, v2 같은 버전 흔적
    → config에서 on/off 가능한 기능들
```

## 3. 문서 구조 (이 순서대로 작성하라)

### 3-1. 프로젝트 한 줄 정의

```
[무슨 문제]를 [어떻게] 해결한 [무엇]
```

코드의 목적을 한 문장으로 정의한다.

**작성 기준:**
- 15단어 이내
- 기술 스택을 나열하지 않는다
- 구현 디테일(검색 방식명, 모델명 등)을 넣지 않는다
- **문제와 결과**만 넣는다

**예시:**
- ✅ "Obsidian 지식 저장소의 검색 누락과 근거 부족을 줄인 로컬 RAG 시스템"
- ❌ "요약-원문 결합 검색과 단계형 검증 흐름으로 만든 로컬 RAG 시스템" ← 구현 디테일이 들어감

### 3-2. 배경 및 문제 정의

**작성할 내용:**

```
왜 이 프로젝트가 필요했는가?
기존에는 어떻게 하고 있었는가?
그게 왜 부족했는가?
```

**필수 규칙:**
- **구체적 숫자를 반드시 포함한다** (문서 수, 폴더 수, 컬렉션 수 등 코드에서 파악 가능한 것)
  - ❌ "폴더가 많고 파일 형식이 섞여 있었다"
  - ✅ "프로젝트별 폴더 15개+, 문서 200건 이상이 쌓여 있었다"
- **문제는 "~할 수 있다"가 아니라 "~했다"로 쓴다**
  - ❌ "무관 문서가 섞일 수 있다"
  - ✅ "무관 문서가 섞였다"
- 문제 테이블의 "구체적 상황"에도 가능성이 아닌 실제 상황으로 쓴다
  - ❌ "타 프로젝트 문서가 상위에 섞일 수 있다"
  - ✅ "타 프로젝트 문서가 상위에 섞였다"

**코드에서 찾는 방법:**
- 입력 데이터의 형태를 보면 어떤 상황인지 유추 가능
- 처리 로직의 복잡도를 보면 단순 검색으로는 안 되는 이유를 설명 가능
- 데이터 전처리 코드가 있다면 원본 데이터의 문제점을 유추 가능
- jobs.yaml, config 파일, 컬렉션 수 등에서 규모 수치를 추출하라

**작성 형식:**
```markdown
## 배경

[2~3문장. 구체적 숫자 포함]

## 문제

[각 문제를 단정형으로 서술]

| 문제 | 구체적 상황 |
|------|------------|
| ... | [실제 발생한 상황으로. "~할 수 있다" 금지] |
```

### 3-3. 기술적 접근

**이 섹션이 문서의 핵심이다.**

**코드에서 추출할 것:**

```
1. 전체 파이프라인을 단계별로 분리하라
   - 코드의 함수/모듈 구조를 따라가면 자연스럽게 나온다

2. 각 단계에서 어떤 기술적 선택을 했는지 파악하라
   - 어떤 모델을 쓰는지
   - 어떤 파라미터를 설정했는지
   - 어떤 전처리를 하는지

3. 그 선택의 이유를 코드 맥락에서 추론하라
   - config에 여러 옵션이 있으면 → 실험을 거쳐 선택한 것
   - 주석이 있으면 → 의도가 담긴 것
   - 복잡한 처리가 있으면 → 단순 방식이 안 됐기 때문
```

**작성 형식 — 각 기술적 선택마다:**

```markdown
### [단계명 또는 개선 포인트]

**문제:**
[이 단계에서 어떤 문제가 있었는지. 단정형으로.]

**접근:**
[어떤 선택지가 있었고, 뭘 골랐는지]

**선택 이유:**
[왜 이걸 골랐는지. 코드에서 근거를 찾아서 쓴다]

**구현:**
[코드에서 실제로 어떻게 구현되어 있는지 핵심만 간결하게]

**결과:**
[이 개선으로 뭐가 달라졌는지. 수치가 있으면 수치로]
```

**코드 인용 기준:**
- 전체 코드를 붙이지 않는다
- 핵심 로직만 5~15줄 이내로 발췌한다
- 코드보다 **코드가 하는 일의 설명**이 중심이다
- 코드 블록을 쓸 때는 반드시 위아래에 맥락 설명을 붙인다

### 3-4. 시스템 구조

**이 섹션의 목적:**
전체 파이프라인의 흐름을 보여주되, README에 있는 기술 스택 나열을 반복하지 않는다.

**쓸 것:**
- 파이프라인 흐름도 (텍스트 기반) — 데이터가 어떻게 흘러가는지
- 각 단계가 왜 이 순서인지 (순서의 이유가 있으면)
- 기술을 언급할 때는 반드시 **선택 이유와 함께**
  - ❌ "임베딩: BAAI/bge-m3"
  - ✅ "다국어 지원과 로컬 실행이 필요해서 bge-m3를 선택했다"

**쓰지 않을 것:**
- 기술 스택 전체 나열 표 (README에 이미 있다)
- 프로젝트 디렉토리 구조 (README에 이미 있다)
- 상세 API 스펙 (README에 이미 있다)

**작성 형식:**
```markdown
## 시스템 구조

```
[입력]
  → 단계 1: [무엇을 하는지]
  → 단계 2: [무엇을 하는지]
  → ...
[출력]
```

[각 단계의 순서가 왜 이렇게 되어야 하는지, 기술 선택이 왜 이것인지를 서술]
```

### 3-5. 결과 및 성능

**이 섹션의 핵심 규칙:**

> **config 값(threshold, retry 횟수, k값, chunk 크기 등)은 결과가 아니다.**
> **config 값은 "기술적 접근"에서 이미 언급되었다.**
> **결과 섹션에는 "그래서 뭐가 달라졌는가"만 쓴다.**

**반드시 포함할 내용:**

#### 1) 정성적 개선 (before/after 비교 필수)

```markdown
| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| [문제 항목] | [이전에는 어땠는지] | [지금은 어떻게 바뀌었는지] |
```

- "개선 전"에는 단일 검색, 기본 구조 등 baseline 상태를 쓴다
- "개선 후"에는 현재 시스템의 실제 동작을 쓴다
- **"~할 수 있다" 금지. "~했다" / "~된다"로 쓴다**

#### 2) 운영 지표

- 코드에 로깅/측정 구조가 있으면 그 필드를 명시한다
- 실측값이 있으면 실측값을 쓴다
- **실측값이 없으면 "미측정"이라고 솔직하게 쓴다. 이건 신뢰를 준다**

#### 3) 정량 평가 계획 (아직 없는 경우)

- 어떤 평가셋을 만들 계획인지
- 어떤 지표를 쓸 계획인지
- 목표 기준선은 얼마인지
- **계획을 구체적으로 쓴다. "추후 평가 예정"처럼 모호하게 쓰지 않는다**

### 3-6. 설계 판단 요약

프로젝트에서 내린 주요 기술적 판단을 표로 정리한다.

```markdown
## 주요 설계 판단

| 문제 상황 | 선택지 | 선택 | 이유 |
|-----------|--------|------|------|
| [뭐가 문제였는지] | [A / B / C] | [뭘 골랐는지] | [왜] |
```

**코드에서 찾는 방법:**
- config 파일에서 설정 가능한 옵션들 → 여러 선택지 중 하나를 고른 흔적
- 주석 처리된 코드 → 이전에 시도했다가 바꾼 흔적
- TODO 주석 → 아직 해결하지 않은 문제
- 조건분기(if/else) → 여러 방식을 지원하도록 설계한 흔적

### 3-7. 한계 및 개선 방향

```markdown
## 현재 한계

- [한계 1]: [왜 한계인지] → [개선 방향]
```

**코드에서 찾는 방법:**
- TODO 주석
- 하드코딩된 값 → 유연하지 않은 부분
- 에러 처리에서 pass나 단순 print로 넘기는 부분 → 미완성
- 지원하지 않는 입력 형식 → 확장 가능성
- 성능 병목이 될 수 있는 동기 처리 → 비동기 전환 가능성

**주의:**
- 한계를 쓰는 건 약점이 아니다
- **"이 부분의 한계를 인지하고 있고, 어떻게 개선할지 알고 있다"**를 보여주는 것이다
- 한계만 쓰지 말고 반드시 **개선 방향**도 함께 쓴다

### 3-8. 회고

**이 섹션은 사고의 깊이를 보여주는 곳이다.**

**필수 규칙:**
- **각 항목을 최소 3~5문장으로 쓴다. 한 줄로 끝내지 않는다**
- 무엇이 어려웠는지 → 왜 어려웠는지 → 어떻게 해결했는지 → 그 과정에서 뭘 알게 되었는지 흐름으로 쓴다
- 코드에서 근거를 찾는다 (구조 변경 흔적, 리팩토링 흔적, workaround 코드)

**작성 형식:**

```markdown
## 회고

### 가장 어려웠던 기술적 판단
[3~5문장. 무엇이 어려웠고, 왜 어려웠고, 어떤 시행착오를 거쳤고, 최종적으로 어떻게 결정했는지]

### 처음 가정과 달랐던 부분
[3~5문장. 처음에 뭘 가정했고, 실제로 돌려보니 뭐가 달랐고, 그래서 뭘 바꿨는지]

### 다시 만든다면 바꿀 부분
[3~5문장. 뭘 바꿀지, 왜 바꿀지, 그러면 뭐가 달라질지]
```

**나쁜 예시 (한 줄짜리):**
```
- 가장 어려웠던 판단: 검색 결합 방식 선택이었다.
```

**좋은 예시 (흐름이 보이는):**
```
요약 문서 검색과 원문 검색을 어떻게 결합할지가 가장 오래 걸렸다.
요약만 검색하면 속도는 빠르지만 원문 위치 연결이 약해졌다.
원문을 직접 검색하면 근거는 강해지지만 chunk 수가 많아져서 노이즈가 늘었다.
결국 요약으로 후보를 좁히고 원문으로 근거를 보강하는 2단계 구조로 결정했다.
이 판단을 내리기까지 단일 검색기로 여러 번 실패한 후였다.
```

### 3-10. 시각 자료 배치 규칙 (GIF/MP4/PNG/JPG)

시각 자료의 목적은 "예쁘게 보이기"가 아니라, 독자가 문서를 읽는 순서에서 이해 속도를 높이는 것이다.

**배치 기준은 '독자 질문' 순서다:**
1. **무엇을 만들었나?**  
   위치: 프로젝트 한 줄 정의 바로 아래  
   자료: 최종 결과 `PNG/JPG` 1장  
2. **무엇이 문제였나?**  
   위치: 배경/문제 정의 끝  
   자료: 문제 재현 `PNG/JPG` 1장  
3. **어떤 핵심 동작이 차별점인가?**  
   위치: 기술적 접근의 핵심 단계 직후  
   자료: 짧은 `MP4` 1개 (10~25초)  
4. **그래서 실제로 얼마나 달라졌나?**  
   위치: 결과/성능 표 바로 아래  
   자료: before/after 비교 `PNG/JPG` 1세트 또는 짧은 `GIF` 1개  
5. **회고/설계 판단 섹션은 왜 이미지가 없나?**  
   이유: 이 구간은 추론과 판단 설명 구간이라 텍스트 집중이 더 효과적이다

**밀도 규칙:**
- 한 섹션당 시각 자료 최대 1개
- 문서 전체 시각 자료 3~4개면 충분
- 동일 장면의 이미지+영상 중복 금지

**형식 규칙:**
- 카드(`header.teaser`)는 정적 이미지(`PNG/JPG/WebP`)만 사용
- 본문 데모는 `MP4` 우선, `GIF`는 보조
- 모든 시각 자료에는 바로 아래 1문장 캡션을 붙인다

**레포 적용 규칙:**
- 이미지: `/assets/images/portfolio/`
- 영상: `/assets/videos/`
- 카드 연결: `_portfolio/*.md`의 `header.teaser`, `frontend_url`

## 4. 최종 체크리스트

문서 작성 후 아래 항목을 **모두** 확인하라:

```
문체/톤:
  □ "~했습니다" 체가 아닌 "~했다" 체로 통일되었는가
  □ "~할 수 있다", "~될 수 있다" 같은 가능성 표현이 없는가
  □ "다양한", "혁신적인", "활용하여" 같은 표현이 없는가
  □ "좋은 경험이었다" 류의 감상이 없는가
  □ 한 문장에 정보가 하나만 있는가

한 줄 정의:
  □ 15단어 이내인가
  □ 구현 디테일(모델명, 검색 방식명)이 아닌 문제+결과 중심인가

배경/문제:
  □ 구체적 숫자가 포함되었는가 (문서 수, 폴더 수, 컬렉션 수 등)
  □ 문제 서술이 "~했다"로 단정되었는가 ("~할 수 있다" 아닌지)
  □ 문제 테이블의 "구체적 상황"도 단정형인가

결과 섹션:
  □ config 값(threshold, k, retry 등)이 결과로 들어가지 않았는가
  □ before/after 정성적 비교 표가 있는가
  □ 운영 지표가 실측값 또는 솔직한 "미측정"으로 되어 있는가
  □ 정량 평가 계획이 구체적인가 (평가셋 규모, 지표, 목표 기준)

시스템 구조:
  □ 기술 스택 단순 나열 표가 없는가 (README에 이미 있다)
  □ 기술을 언급할 때 선택 이유가 함께 있는가
  □ 디렉토리 구조, API 스펙이 없는가 (README에 이미 있다)

회고:
  □ 각 항목이 최소 3~5문장인가 (한 줄로 끝나지 않았는가)
  □ 사고 흐름이 보이는가 (어려웠던 이유 → 시행착오 → 결론)

코드 인용:
  □ 전체 코드를 붙이지 않고 핵심만 발췌했는가
  □ 코드 블록 위아래에 맥락 설명이 있는가

README 분리:
  □ 설치법, 실행법이 들어가지 않았는가
  □ API 엔드포인트 목록이 없는가
  □ 프로젝트 디렉토리 구조 전체가 없는가
  □ 문서 하단에 GitHub 링크가 있는가

시각 자료:
  □ 시각 자료가 독자 질문 순서(무엇/문제/핵심동작/결과)에 맞게 배치되었는가
  □ 한 섹션당 1개, 문서 전체 3~4개 밀도를 지켰는가
  □ 카드(`header.teaser`)는 정적 이미지, 본문 데모는 MP4 우선으로 구성했는가
  □ 모든 시각 자료에 1문장 캡션이 있는가
  □ GIF가 5MB를 넘지 않는가 (초과 시 MP4 대체)
```

## 5. 절대 하지 말아야 할 것

```
1. 기술 스택만 나열하고 끝내지 마라
   → 기술은 반드시 "왜 이걸 선택했는지"와 함께

2. 코드 전체를 붙이지 마라
   → 코드는 GitHub 링크로 충분하다
   → 문서에는 핵심 로직 발췌만

3. 튜토리얼처럼 쓰지 마라
   → "먼저 ~를 설치합니다. 다음으로 ~를 합니다" ❌

4. 모든 게 완벽하다고 쓰지 마라
   → 한계를 모르는 것처럼 보이면 신뢰가 떨어진다

5. 학습 과정을 쓰지 마라
   → "처음에 이 개념을 몰랐는데 공부해서 알게 되었다" ❌
   → 포트폴리오는 학습 일기가 아니다

6. 결과 섹션에 config 파라미터를 나열하지 마라
   → threshold, k, retry 횟수 등은 "기술적 접근"에서 이미 다뤘다
   → 결과에는 "뭐가 달라졌는가"만 쓴다

7. 가능성으로 쓰지 마라
   → "~할 수 있다"는 겪어보지 않은 사람의 표현이다
   → "~했다"로 써야 경험에서 나온 문제라는 인상이 생긴다

8. README에 이미 있는 내용을 반복하지 마라
   → 기술 스택 표, 설치법, API 목록, 디렉토리 구조는 README의 영역이다
   → 블로그에서는 "왜 이 기술을 골랐는지"만 다룬다
   → 상세 코드와 구조는 하단 GitHub 링크로 연결한다
```

## 6. 최종 문서 목차 요약

```
1. 프로젝트 한 줄 정의 (15단어 이내, 문제+결과 중심)
2. 배경 및 문제 정의 (구체적 숫자, 단정형 서술)
3. 기술적 접근 (핵심 — 가장 길게. 문제→접근→이유→구현→결과)
4. 시스템 구조 (파이프라인 흐름도. 기술 선택 이유 중심. 스택 나열은 README 참조)
5. 결과 및 성능 (before/after 비교 + 운영 지표 + 평가 계획. config 나열 금지)
6. 주요 설계 판단 (선택지 비교 표)
7. 한계 및 개선 방향 (한계 + 반드시 개선 방향)
8. 회고 (각 항목 3~5문장, 사고 흐름)
9. 코드 링크 (GitHub Repository)
10. 시각 자료 배치 점검 (독자 질문 순서 기준)
```
