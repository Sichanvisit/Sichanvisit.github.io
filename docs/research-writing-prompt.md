# Research 정리 기준과 생성 프롬프트

## 목적

이 문서는 `research` 섹션을 어떤 방식으로 정리하는지 설명하고, 앞으로 ML / DL / LLM 노트를 같은 톤과 구조로 재정리하기 위한 기준 프롬프트를 남기기 위해 작성했다.

핵심 목표는 단순하다.

- 제목과 실제 내용이 맞아야 한다.
- 설명보다 구현 흔적이 먼저 보여야 한다.
- 원본 노트의 큰 흐름은 살리되, `EDA`, `결측치 확인`, `환경준비` 같은 저신호 구간이 글 앞부분을 점령하지 않게 해야 한다.
- 방문자가 "무엇을 공부했고", "무엇을 직접 해봤고", "어떤 코드가 남아 있는지"를 빠르게 파악할 수 있어야 한다.

---

## 현재 Research 정리 원칙

### 1. 제목과 직접 맞닿는 내용을 앞으로 둔다

`research` 글은 제목과 연결되는 구현 주제가 앞에 나와야 한다.

- `XGBoost` 글이면 `XGBoost 회귀`, `XGBoost 분류`, 모델 코드, 학습/평가 코드가 먼저 보여야 한다.
- `AlexNet` 글이면 모델 구조, 추론 흐름, 전처리 변환, 결과 해석이 먼저 보여야 한다.
- `임베딩 비교` 글이면 전처리 비교, 임베딩 실험, 학습 준비, 결과 비교가 먼저 보여야 한다.

반대로 아래 내용은 제목과 직접 연결되지 않으면 뒤로 밀린다.

- 환경준비
- import 모음
- GPU 설정
- 한글 폰트 설치
- 단순 `head()`, `info()`, `describe()`
- 결측치 / 중복값 / 이상치 점검만 반복하는 구간

### 2. 원본 노트의 큰 장은 살리되, 다시 큐레이션한다

원본 `.md` / `.ipynb`의 `# / ## / ###` 구조는 최대한 보존한다. 다만 research 글에서는 그대로 복사하지 않고, "지금 이 글에서 먼저 보여줄 가치가 있는 장"을 다시 고른다.

즉,

- 원본 구조 보존
- 보여주는 순서는 재정렬
- 장 제목은 최대한 유지
- 세부 설명은 원문에서 요약
- 코드 블록은 핵심 구현 중심으로 재선정

### 3. 글은 "설명형 아카이브"가 아니라 "구현형 학습 기록"이어야 한다

좋은 research 글은 아래 질문에 답해야 한다.

- 이 글에서 다룬 핵심 주제가 무엇인가
- 어떤 구현 단계가 있었는가
- 어떤 코드가 실제로 남아 있는가
- 실험/비교/평가를 어디까지 해봤는가

그래서 research 글은 보통 아래 순서를 따른다.

1. `글 한눈에 보기`
2. `원본 노트 흐름` 또는 `What This Note Covers`
3. `구현 흐름` 또는 `Implementation Flow`
4. `코드로 확인한 내용` 또는 `Code Highlights`
5. `참고 자료` / `Source Bundle`

### 4. 코드 블록은 "대표 구현 장면"으로 뽑는다

코드는 많다고 좋은 게 아니라, 대표성이 있어야 한다.

우선순위는 대체로 아래 순서다.

1. 모델 정의
2. 학습 루프
3. 평가 코드
4. 전처리 / 피처 엔지니어링
5. 데이터 로딩
6. 단순 환경설정

즉 `pip install`, `drive.mount`, `font install`보다는

- `XGBRegressor`, `XGBClassifier`
- `Dataset`, `DataLoader`
- `Bi-LSTM`
- `UNet`, `Mask R-CNN`
- `Retriever`, `VectorStore`, `LoRA`

같은 코드가 먼저 보여야 한다.

## Track별 정리 기준

## ML

ML 글은 아래 흐름이 잘 보이면 된다.

- 문제 정의
- 데이터 전처리
- 피처 엔지니어링
- 모델링
- 학습 / 튜닝
- 평가 / 결과 저장

앞쪽에 잘 보여야 하는 키워드 예시:

- `LinearRegression`, `LogisticRegression`
- `DecisionTree`, `RandomForest`, `XGBoost`
- `GridSearchCV`
- `StandardScaler`, `LabelEncoder`
- `RMSLE`, `F1`, `RMSE`

## DL

DL 글은 아래 흐름을 우선한다.

- 모델 구조
- 입력 파이프라인
- 학습 루프
- 추론 / 시각화
- 결과 해석

앞쪽에 잘 보여야 하는 키워드 예시:

- `AlexNet`, `ResNet`, `CNN`
- `UNet`, `Segmentation`
- `Mask R-CNN`, `Faster R-CNN`
- `DataLoader`, `loss`, `optimizer`
- `inference`, `augmentation`, `forward`

## LLM

LLM 글은 아래 흐름을 우선한다.

- 전처리 / 토큰화
- 임베딩 또는 모델 준비
- 체인 / 검색 / 파이프라인 구성
- 실험 비교
- 평가 / 결과 요약

앞쪽에 잘 보여야 하는 키워드 예시:

- `Word2Vec`, `FastText`, `GloVe`
- `LSTM`, `GRU`
- `RAG`, `Retriever`, `VectorStore`
- `LoRA`, `QLoRA`
- `LangChain`, `LangGraph`, `Agent`

---

## 작성할 때 피해야 하는 것

- 제목과 상관없는 `EDA`, `결측치`, `이상치` 설명이 본문 앞을 점유하는 것
- 원문 문장을 길게 붙여서 무엇을 말하고 싶은지 흐려지는 것
- 코드보다 설명이 훨씬 많은 것
- `환경준비`, `import`, `설치`, `다운로드`가 대표 구현처럼 보이는 것
- 실험 결과가 있는데도 앞쪽 구조에서 빠지는 것
- 원문에 `모델링`, `학습`, `평가`, `추론`, `실험`이 있는데도 잡지 못하는 것

## 생성 프롬프트

아래 프롬프트는 원본 `.md` / `.ipynb`를 `research`용 글로 재구성할 때 사용하는 기준 프롬프트다.

```text
당신은 AI/ML/DL/LLM 학습 노트를 포트폴리오형 research 글로 재구성하는 편집자다.

목표:
- 원본 노트의 큰 흐름은 보존하되, research 글에서는 제목과 직접 연결되는 구현 내용을 앞에 둔다.
- 방문자가 "무엇을 공부했고", "무엇을 직접 구현했고", "어떤 코드가 남아 있는지"를 빠르게 이해하게 만든다.
- 설명 위주가 아니라 구현 위주로 정리한다.

입력:
- track: ML / DL / LLM
- title: 최종 research 제목
- source_title: 원본 노트 제목
- source_note_outline: 원본 노트의 # / ## / ### 구조
- source_sections: 각 섹션의 핵심 문단
- code_blocks: 코드 블록 목록
- companion_files: ipynb, md, 기타 파일 목록
- libraries: 사용 라이브러리 목록

절대 규칙:
1. 제목과 직접 관련된 구현 내용을 맨 앞에 둔다.
2. 환경설정, import, GPU 설정, 폰트 설치, mount, 단순 EDA는 뒤로 민다.
3. 원문에 모델링/학습/평가/실험/추론/결과가 있으면 반드시 앞부분 구조에 반영한다.
4. 코드 블록은 대표 구현 장면 중심으로 3~6개 뽑는다.
5. 코드 설명은 1~2문장만 붙이고, 코드 자체가 중심이 되게 한다.
6. 제목과 맞지 않는 일반론은 쓰지 않는다.
7. 원문에 없는 과장된 성과를 만들어내지 않는다.

track별 우선순위:
- ML: 전처리 -> 피처 엔지니어링 -> 모델링 -> 학습/튜닝 -> 평가
- DL: 모델 구조 -> 입력 파이프라인 -> 학습 루프 -> 추론/시각화 -> 결과
- LLM: 전처리/토큰화 -> 임베딩/모델 준비 -> 체인/검색/실험 -> 평가/결론

출력 형식:

## 글 한눈에 보기
- 문제 설정:
- 원본 구조:
- 데이터 맥락:
- 구현 흐름:
- 자료:
- 주요 스택:

## 원본 노트 흐름 또는 What This Note Covers
- 원문에서 실제로 중요한 장 3~6개만 고른다.
- 각 장은 제목 + 2~4문장 설명 + 읽을 포인트로 정리한다.
- 필요하면 #### 하위 장 2~3개를 붙인다.

## 구현 흐름 또는 Implementation Flow
1. 실제 구현 순서를 4~6단계로 정리한다.
2. 각 단계는 "무엇을 했는지"가 분명해야 한다.
3. 단순 EDA보다 모델/실험/평가 단계가 앞에 와야 한다.

## 코드로 확인한 내용 또는 Code Highlights
- 대표 코드 블록 3~6개
- 각 코드 블록마다:
  - 제목
  - 직접 해본 단계
  - 핵심 API 또는 핵심 개념
  - 1~2문장 설명
  - 코드

## 참고 자료 또는 Source Bundle
- source path
- source formats
- companion files
- note type
- updated_at
- related notes

스타일:
- 한국어로 작성한다.
- 문장은 짧고 분명하게 쓴다.
- 원문을 길게 복붙하지 말고 요약한다.
- "무엇을 공부했는지"보다 "어떤 구현을 직접 해봤는지"가 더 잘 보이게 쓴다.
- 제목과 직접 관계없는 결측치/EDA 설명은 앞에 두지 않는다.
```

## 출력 템플릿

실제 글을 만들 때는 아래 골격을 사용한다.

```md
## 글 한눈에 보기

- 문제 설정:
- 원본 구조:
- 데이터 맥락:
- 구현 흐름:
- 자료:
- 주요 스택:

## 원본 노트 흐름

### [핵심 장 1]

[이 장에서 무엇을 구현했는지 2~4문장]

- 읽을 포인트: [핵심 하위 흐름]

#### [하위 장]

[원문 핵심 요약]

## 구현 흐름

### 1. [단계 이름]

- 단계:
- 구현 의도:
- 핵심 API:
- 코드 포인트:

## 코드로 확인한 내용

### [코드 제목]

**직접 해본 단계**: [단계]

**핵심 API**: `[API]`

[코드 설명 1~2문장]

```python
# representative code
```
```

## 사용 원칙 한 줄 요약

`research`는 "배운 이론 설명문"이 아니라, `제목과 직접 연결되는 구현 흐름 + 남겨진 코드 증거`를 보여주는 학습 아카이브로 정리한다.
